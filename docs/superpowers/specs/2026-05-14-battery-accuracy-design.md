# 电池电量显示准确性提升设计

## 概述

当前项目在 `src/main.cpp` 中直接使用 `M5.Power.getBatteryVoltage()` 读取电池电压，并将 `3300-4100mV` 线性映射到 `0-100%`，再做简单 EMA 平滑。这种做法实现简单，但会带来两个明显问题：

- 百分比与真实剩余续航不一致，因为锂电放电曲线不是线性的
- 电压受瞬时负载、充放电切换影响较大，显示值容易出现不符合直觉的跳动

本设计目标是在不要求用户专门做一次完整标定的前提下，通过“默认基线曲线 + 日常使用自动学习 + 显示稳定策略”提升电池电量显示的准确性与稳定性。

## 目标

- 百分比显示比当前线性映射更接近日常真实剩余续航
- 显示值比当前实现更稳定，避免短时间来回跳动
- 冷启动阶段允许先使用基线估算，并在后续日常使用中逐步变准
- 学习过程自动进行，不要求用户手工记录完整充放电曲线
- 学习数据异常或缺失时能够回退到默认基线，不影响基本显示

## 非目标

- 不提供实验室意义上的精确 SOC 估算
- 不要求首版显示“剩余可用小时数”
- 不实现用户手工校准流程
- 不做完整历史时间序列回放或复杂拟合模型
- 不扩展新的 UI 页面，仅替换现有电量百分比来源

## 用户约束与结论

根据本次确认，方案边界如下：

- 优先提升“百分比更接近真实剩余电量”和“数值更稳定”
- 不做专门标定，依靠日常使用自动学习
- 目标含义定义为“这台设备在用户日常使用方式下的剩余续航百分比”
- 允许冷启动阶段先使用估算值，随着使用逐步变准

因此本设计采用“基线曲线 + 自动学习”的渐进式方案，而不是一次性设备标定或纯电压线性映射。

## 架构

### 新增模块

- `src/battery_estimator.h` - `BatteryEstimator` 类声明、数据结构定义
- `src/battery_estimator.cpp` - 电池采样处理、基线估算、自动学习、显示稳定逻辑

### 修改模块

- `src/main.cpp` - 用 `BatteryEstimator` 替换当前电量显示分支里的线性映射逻辑
- `src/config_store.h` - 声明电池学习数据的加载与保存接口
- `src/config_store.cpp` - 实现电池学习数据的持久化读写

电池学习相关结构体统一定义在 `src/battery_estimator.h`，不放入现有 `src/config_types.h`，避免与 Wi-Fi / 时区配置模型耦合。

### 分层职责

电量显示拆分为三层，避免继续把算法堆在 `main.cpp` 的显示分支中。

1. 原始采样层
   - 读取 `M5.Power.getBatteryVoltage()`
   - 读取 `M5.Power.isCharging()`

首版主估算只使用“电压 + 是否充电”两个输入，不把 `M5.Power.getBatteryCurrent()` 纳入主模型，避免因板级噪声或采样不稳定扩大实现复杂度。

2. 估算层
   - 根据默认锂电曲线计算 `baselinePercent`
   - 根据学习到的设备特征修正为更贴近续航的 `runtimePercent`

3. 显示层
   - 对最终结果做限速、去抖、充放电状态约束
   - 输出稳定的整数百分比供 `drawBattery(...)` 使用

## 数据流

每次电池采样的处理流程如下：

1. 定期读取电池原始数据
2. 对原始电压做毛刺过滤和平滑
3. 根据默认放电曲线得到冷启动可用的基线百分比
4. 如果已有足够历史样本，则用学习参数对基线百分比做修正
5. 通过显示稳定器得到最终 UI 百分比
6. 将当前样本送入学习缓存，判断是否满足更新学习参数的条件
7. 在满足条件时保存新的学习结果

这样主循环只负责喂入采样值和读取显示结果，不再直接处理映射细节。

## BatteryEstimator 设计

### 目标职责

`BatteryEstimator` 只负责“输入电池状态，输出稳定的显示百分比与学习状态”。它不负责绘制 UI，也不直接处理与电池无关的配置逻辑。

### 接口

```cpp
struct BatterySample {
    int voltageMv;
    bool charging;
    unsigned long timestampMs;
};

struct BatteryEstimate {
    int displayPercent;
    float baselinePercent;
    float runtimePercent;
    bool charging;
    bool usingLearnedProfile;
};

class BatteryEstimator {
   public:
    void begin();
    void loadProfile(const BatteryProfile& profile);
    BatteryProfile exportProfile() const;

    BatteryEstimate update(const BatterySample& sample);

   private:
    // 内部状态与学习参数
};
```

接口最终命名可以按项目现有风格微调，但职责边界应保持不变。

## 学习数据设计

首版不存整段电压时间序列，仅存少量可解释、可控的学习参数，避免复杂度和写入成本过高。

### 持久化字段

```cpp
struct BatteryProfile {
    uint16_t learnedEmptyMv;
    uint16_t learnedFullMv;
    int16_t curveBias;
    float avgDischargeRate;
    uint16_t sampleCount;
    bool initialized;
};
```

其中：

- `learnedEmptyMv`
  表示这台设备在用户日常使用模式下接近低电量时的代表电压

- `learnedFullMv`
  表示这台设备在充满并稳定后可视为高电量锚点的代表电压

- `curveBias`
  用于修正默认曲线的中段映射，让百分比更贴近这台设备的实际续航体验

- `avgDischargeRate`
  表示在日常使用模式下的平均掉电速度，用于把基础电量修正为更像“剩余续航百分比”的结果

- `sampleCount`
  表示学习样本量，用于控制学习结果权重

- `initialized`
  标记当前 profile 是否已有有效学习结果

## 自动学习规则

### 1. 满电锚点学习

当设备经历持续充电，并在充电结束后进入高电压稳定区时，更新 `learnedFullMv`。

设计要求：

- 不能在单次读到高电压时立即认定满电
- 需要充电结束后的稳定窗口，避免表面电压偏高导致学习失真
- 每次更新幅度受限，防止单次异常样本把高电压锚点拉偏

### 2. 低电锚点学习

当设备持续处于放电状态，且长时间保持在低电压区时，更新 `learnedEmptyMv`。

设计要求：

- 不要求设备一定放到自动关机才学习
- 低电锚点应代表“用户实际认为接近没电”的工作区，而不是理论绝对空电点
- 学习值必须被限制在合理电压范围内

### 3. 放电速率学习

只在“有效日常放电片段”中学习平均放电速度。

有效片段至少满足：

- 当前不在充电
- 持续时间足够长
- 电压变化方向与放电一致
- 中间没有明显异常跳变
- 样本区间不处于刚插拔电源的过渡窗口

通过这些片段更新 `avgDischargeRate`，让估算从“还有多少电”逐步接近“还能用多久”。

### 4. 曲线偏置学习

默认曲线负责提供冷启动能力，长期使用过程中再通过 `curveBias` 做小幅修正。

首版只做简单偏置，不做高阶多项式拟合或完整曲线重建，原因如下：

- 项目当前需求不需要复杂模型
- 简单偏置更容易解释、调试和限制范围
- 即使学习效果一般，也能平稳退回默认基线

## 防止学坏的约束

自动学习必须有明确边界，避免系统随着使用越来越偏。

约束如下：

- 只对高质量样本进行学习
- 单次学习更新幅度受限
- 学习参数必须落在合理物理范围内
- 当样本量不足时，优先使用默认基线
- 检测到自相矛盾的学习结果时，直接丢弃本次更新

例如：

- 若 `learnedFullMv <= learnedEmptyMv`，则视为无效结果
- 若新学习值超出预设上下界，则不写入 profile
- 若样本片段包含明显反向跳变，则不用于更新平均放电速度

## 默认基线估算

当前线性映射会被替换为基于锂电放电特性的分段或曲线映射。该基线估算的职责是：

- 在没有学习数据时提供合理冷启动结果
- 在学习数据不足时作为主导结果
- 在学习数据异常时作为安全回退结果

首版不要求一次就获得极致准确的默认曲线，但必须明显优于 `3300-4100mV` 的线性映射。

## 显示稳定策略

估算器内部结果和 UI 显示结果分离处理。

### 1. 内部值连续，显示值限速

- 内部估算可以保留连续变化
- UI 仅显示整数百分比
- 放电时不允许短时间大幅下降
- 小范围抖动不立即传递到显示层

### 2. 充电与放电分状态处理

- 放电时默认单调不升，除非进入新的高可信区间
- 充电时默认单调不降，避免插着电反而掉电量
- 刚插入或拔出充电线后，进入短暂观察窗口，不立刻相信瞬时电压

### 3. 显示死区

如果新旧结果差异很小，则不更新显示值。这样可以减少用户感知到的抖动，也避免采样窗口过短时引入的 UI 频繁变化。

### 4. 低电区保守处理

低电量区优先保证稳定和可预期，不允许短时间出现连续大跳水。该区间的估算应更保守，并适当降低瞬时样本权重。

## 异常场景处理

### 刚拔充电线

- 电压可能短时虚高
- 不能立刻据此上调显示百分比
- 应在过渡窗口中逐步向新的放电估算收敛

### 短时高负载或瞬时掉压

- 瞬时压降只作为观测样本，不直接驱动显示下降
- 仅当低电压持续一段时间后，才认为这是有效掉电

### 重启恢复

- 启动时恢复已保存的学习参数
- 首次采样后重新建立平滑状态和显示状态
- 首版不持久化运行期瞬时状态，避免频繁写入 NVS

### 学习数据不足

- 当 `sampleCount` 过低时，系统应明确处于“基线优先”模式
- 随着样本积累，再逐步提高学习结果的权重

### 学习结果异常

- 学到的参数不合理时，保留旧 profile
- 如果旧 profile 也无效，则退回默认基线

## 持久化设计

### 原则

- 电池学习数据与 `DeviceConfig` 分离
- 不把自动学习状态耦合进现有 Wi-Fi / 时区配置保存流程
- 控制写入频率，只在关键学习结果更新时保存

### 接口

```cpp
class ConfigStore {
   public:
    bool load(DeviceConfig& outConfig);
    bool save(const DeviceConfig& config);

    bool loadBatteryProfile(BatteryProfile& outProfile);
    bool saveBatteryProfile(const BatteryProfile& profile);
};
```

## main.cpp 集成方式

当前 `DisplayMode::Battery` 分支中存在如下职责混杂：

- 采样频率控制
- 原始电压平滑
- 电压到百分比映射
- 显示字符串格式化

首版集成后，`main.cpp` 改为：

1. 定时读取当前电池样本
2. 将样本交给 `BatteryEstimator::update(...)`
3. 取回 `displayPercent`
4. 格式化为 `"%d%%"`
5. 调用现有 `drawBattery(...)`

也就是说，`main.cpp` 只保留调用流程，不保留估算细节。

## 调试支持

首版包含一个默认关闭的串口调试开关，用于硬件验证阶段输出：

- 当前电压
- 当前是否充电
- 基线百分比
- 学习修正后的百分比
- 最终显示百分比
- 当前样本量

这样在真实设备上可以更快判断偏差来自：

- 默认基线不准
- 学习参数更新过慢或过快
- 显示稳定器过于激进

## 测试策略

项目已有 native 测试体系，首版应以纯逻辑单元测试为主。

### 新增测试

- `test/native/test_battery_estimator/test_main.cpp`

### 1. 冷启动基线测试

验证在没有学习数据时：

- 输出始终落在 `0-100%`
- 基线结果符合预期的分段/曲线区间
- 对同一组典型电压点，比当前线性映射更符合锂电放电直觉

### 2. 显示稳定测试

构造带抖动的电压序列，验证：

- 显示值不会来回跳动
- 放电时不会轻易回升
- 充电时不会轻易回落
- 插拔电源后的过渡窗口不会出现明显反直觉跳变

### 3. 学习更新测试

构造模拟日常使用片段，验证：

- 满电锚点会在有效样本下更新
- 低电锚点会在有效样本下更新
- 放电速率会逐步修正
- 异常样本不会污染已有 profile
- 样本量不足时仍以基线为主

### 4. 持久化测试

在 native 环境验证：

- `BatteryProfile` 可正确保存与读取
- 缺失 profile 时能回退到默认值
- 异常 profile 数据不会导致估算器进入无效状态

## 硬件验证

虽然首版重点是单元测试，但仍需要基本的真机验证：

1. 连续使用数天，观察显示值是否比旧版本更符合主观续航感受
2. 在插线充电、拔线放电两个切换点，确认不会出现明显跳变
3. 在中高电量区轻微负载波动时，确认显示不来回抖动
4. 在低电量区确认不会短时间大幅跳水

## 验收标准

- 冷启动时，电量显示不比当前版本更抖
- 相同抖动输入下，显示跳动次数明显少于当前版本
- 日常使用数天后，显示百分比逐步更贴近用户主观续航感受
- 低电量区不出现短时间连续大跳水
- 充电或拔线后不出现明显反直觉跳变
- 学习数据损坏、缺失或异常时，系统能回退到默认基线

## 实施边界

本设计只覆盖以下内容：

- 新增电池估算模块
- 引入默认基线曲线
- 引入自动学习参数与持久化
- 引入显示稳定器
- 补充 native 单元测试与基本硬件验证

本设计不包含：

- 展示剩余小时数
- 用户手工校准页面或调试 UI
- 复杂机器学习模型
- 面向不同使用场景的多 profile 切换
